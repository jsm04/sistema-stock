# Primer sprint

La selección de **UC01 (gestión de productos), UC02 (consulta de inventario) y UC07 (usuarios)** responde a una dependencia estructural: no es viable operar stock sin antes tener entidades base (productos) y un mecanismo mínimo de acceso (usuarios). UC02 aporta visibilidad, que es necesaria incluso antes de introducir movimientos de stock. En términos de valor incremental, este recorte permite un sistema usable aunque todavía incompleto.

---

## Refinamiento del alcance del sprint

Se mantiene deliberadamente simple, evitando lógica innecesaria.

### UC01 – Gestionar productos (refinado)

Alcance:

- Crear producto
- Editar producto (campos básicos)
- Consultar producto (por ID)

Se excluye:

- Eliminación física (se evita por ahora para no introducir soft delete)

Restricciones:

- SKU único
- Stock inicial ≥ 0
- Precio ≥ 0

---

### UC02 – Consultar inventario (refinado)

Alcance:

- Listado de productos
- Filtro por nombre (LIKE simple)
- Filtro por categoría exacta

Se excluye:

- Paginación (solo si el dataset crece)
- Ordenamientos complejos

---

### UC07 – Gestión de usuarios (refinado)

Alcance:

- Crear usuario
- Login básico (usuario + contraseña)
- Rol simple: `admin` | `operador`

Se excluye:

- Recuperación de contraseña
- Permisos granulares

Restricciones:

- Username único
- Password hasheado (aunque sea implementación simple)

---

## Escenarios por caso de uso

### UC01 – Crear producto

Escenario básico:

- Usuario autenticado ingresa nombre, SKU, precio, stock
- Sistema valida constraints
- Sistema persiste producto

Escenario alternativo:

- SKU ya existe → rechazo con error

Escenario alternativo:

- Datos inválidos (precio negativo) → rechazo

---

### UC01 – Editar producto

Escenario básico:

- Usuario selecciona producto
- Modifica nombre/precio/categoría
- Sistema guarda cambios

Escenario alternativo:

- Producto inexistente → error

---

### UC02 – Listar productos

Escenario básico:

- Usuario solicita listado
- Sistema devuelve todos los productos

Escenario con filtro:

- Usuario envía `name=“lap”`
- Sistema aplica LIKE y devuelve coincidencias

Escenario con categoría:

- Usuario filtra por categoría exacta
- Sistema devuelve subconjunto

---

### UC02 – Ver detalle de producto

Escenario básico:

- Usuario solicita producto por ID
- Sistema devuelve datos completos

Escenario alternativo:

- ID inexistente → error

---

### UC07 – Crear usuario

Escenario básico:

- Admin ingresa username, password, rol
- Sistema valida unicidad
- Sistema guarda usuario

Escenario alternativo:

- Username duplicado → error

---

### UC07 – Login

Escenario básico:

- Usuario ingresa credenciales
- Sistema valida password
- Sistema retorna sesión/token

Escenario alternativo:

- Credenciales inválidas → error

---

## Planificación del sprint (breve y operativa)

Duración estimada: 1 sprint corto (5–7 días)

### Orden de implementación (dependencias técnicas)

1. **Infraestructura**
   - Configuración Drizzle + SQLite (ya definida)
   - Definición de schemas: `products`, `users`

2. **UC07 (usuarios primero)**
   - Modelo de usuario
   - Registro
   - Login (sin JWT complejo, puede ser sesión simple)

3. **UC01 (productos)**
   - CRUD parcial (create, update, get by id)
   - Validaciones con Zod

4. **UC02 (consulta)**
   - Listado
   - Filtros básicos

---

### Distribución en tareas (granular)

- T1: Schema `users` + migración
- T2: Endpoint crear usuario
- T3: Endpoint login
- T4: Schema `products` + migración
- T5: Endpoint crear producto
- T6: Endpoint editar producto
- T7: Endpoint obtener producto
- T8: Endpoint listar productos + filtros
- T9: Validaciones con Zod en todos los inputs

---

### Criterio de “done”

- Todos los endpoints responden con validación consistente
- No hay violaciones de constraints (SKU, username)
- Base de datos migrada correctamente
- Casos alternativos manejados explícitamente
